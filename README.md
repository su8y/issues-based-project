# Github 기반 프로젝트를 위한 프로젝트 템플릿

이 저장소는 Github Issue 기반 프로젝트를 성공적으로 완수하기 위한 가이드 라인을 제공합니다.

[기여(참여)가이드](contributing.md)
[코딩 규칙 가이드](coding-convention.md)

## 제공하는 것
- github 이슈 및 PR 템플릿
- contributing guide

## 이 템플릿의 목적
이 템플릿은 개인 개발 스타일을 강제하기 위함이 아니라, 팀 단위 개발에서 다음 문제를 해결하기 위해 설계되었다.

- 작업 히스토리의 불명확성
- 리뷰 기준의 개인차
- 신규 인력 온보딩 비용 증가
- 릴리즈 단위 관리의 어려움

## 왜 Github-Issue 기반인 것인가 ?
Issue는 단순한 할 일(To-do)이 아니라, 왜 이 작업이 필요한지에 대한 **공식 기록(Single Source of Truth)** 이다.

Issue에는 다음과 같은 정보가 포함된다:
- 문제의 배경
- 기대 동작
- 논의 히스토리
- 설계 의도
즉, 코드만 봐서는 알수없는 맥락을 명확하게 포함한다.

### 1. 협업시 중복작업 방지
Issue 기반으로 작업할 경우:
- Assignee가 명확히 지정되고
- 작업 진행 상태가 공유되며
- 동일 작업에 대한 중복 개발이 차단된다.

Issue 기반이 아닐 경우 다음 문제가 빈번히 발생한다:
- 동일한 내용을 여러 사용자가 동시에 수정
- PR 충돌 및 중복 리뷰
- “이 작업 누가 하고 있었나요?”라는 커뮤니케이션 비용 발생

### 2.PR 리뷰 품질이 압도적으로 좋아진다
Issue 없이 생성된 PR은 다음과 같은 추가 비용을 발생시킨다:
- 변경 이유 파악
- 요구사항 재확인
- 설계 의도 추론

issue 기반 PR은 변경 이유가 issue에 있고 reviewer는 구현에만 집중해서 리뷰를 할 수 있다.
```
Fixed: #123
```

### 3.프로젝트 관리가 가능해진다 (개발 != 개인 작업)

Issue 기반 프로젝트는 다음이 가능해진다:
- 백로그 관리
- 우선순위 조정
- 릴리즈 단위 묶기
- 진행률 파악

### 4.신규 기여자(Onboarding)가 쉬워진다
모두 신규 기여자의 역할로써 프로젝트의 투입이 된 경험을 가지고 있을것이다.
신규 기여자들의 어려움을 통계로 보았을때 공통적으로 "뭐 부터 해야하는가?"에 대해 어려움을 가진다.

issue 기반 프로젝트는 아래와 같은 내용을 통해 Onboarding의 난이도를 극도로 낮춘다:
- 난이도 명시
- 프로젝트의 큰 그림의 **완벽한 이해**

### 5.기여 이력이 "증명 가능한 기록"이 된다.(HR)
단순 커밋은 무엇을 해결하는 내용인지 불명확하다. 이런 내용만으로는 기여자들이 어떤 기여를 하였고 얼마나 기여를 했는지에 대한 추적이 어렵다.
issue 기반 프로젝트는 `문제 제기 -> 해결 -> 리뷰 -> 머지` 과정의 자동 기록을 통한 인력의 기여율을 추적할 수 있다.

### 6.릴리즈 / 회귀 추적이 쉬워진다
issue 번호를 기준으로 추적이 가능하기 때문에 회귀 추적이 쉬워진다.
- “이 버그 언제 들어왔지?”
- “이 기능 왜 추가됐지?”

### 7.메인테이너/관리자 관점에서 관리 비용이 줄어든다
Issue 기반 프로젝트는:
- PR 거절 기준이 명확
- 논의는 코드 밖에서 해결
- 결정 기록이 남음
