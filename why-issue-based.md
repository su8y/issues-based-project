## 1. 협업시 중복작업 방지
Issue 기반으로 작업할 경우:
- Assignee가 명확히 지정되고
- 작업 진행 상태가 공유되며
- 동일 작업에 대한 중복 개발이 차단된다.

Issue 기반이 아닐 경우 다음 문제가 빈번히 발생한다:
- 동일한 내용을 여러 사용자가 동시에 수정
- PR 충돌 및 중복 리뷰
- “이 작업 누가 하고 있었나요?”라는 커뮤니케이션 비용 발생

## 2. PR 리뷰 품질이 압도적으로 좋아진다
Issue 없이 생성된 PR은 다음과 같은 추가 비용을 발생시킨다:
- 변경 이유 파악
- 요구사항 재확인
- 설계 의도 추론

issue 기반 PR은 변경 이유가 issue에 있고 reviewer는 구현에만 집중해서 리뷰를 할 수 있다.
```
Fixed: #123
```

## 3. 프로젝트 관리가 가능해진다 (개발 != 개인 작업)

Issue 기반 프로젝트는 다음이 가능해진다:
- 백로그 관리
- 우선순위 조정
- 릴리즈 단위 묶기
- 진행률 파악

## 4. 신규 기여자(Onboarding)가 쉬워진다
신규 기여자들이 가장 어려워하는 공통적인 문제는 “무엇부터 시작해야 하는가?”다.

Issue 기반 프로젝트는 다음을 통해 온보딩 난이도를 낮춘다:
- 작업 난이도 명시 및 범위 명시
- 작업의 전체 구조와 흐름에 대한 이해 제공

## 5. 기여 이력이 "증명 가능한 기록"이 된다.(HR)
단순 커밋 기록만으로는 다음을 판단하기 어렵다:
- 무엇을 해결했는지
- 어떤 난이도의 문제였는지
- 협업과 리뷰에 어떻게 참여했는지

issue 기반 프로젝트는 `문제 제기 -> 해결 -> 리뷰 -> 머지`의 전 과정이 자동으로 기록되어
인력의 기여 내용과 기여도를 객관적으로 추적 할 수 있다.

## 6. 릴리즈 / 회귀 추적이 쉬워진다
issue 번호를 기준으로 추적이 가능하기 때문에 회귀 추적이 쉬워진다.
- “이 버그 언제 들어왔지?”
- “이 기능 왜 추가됐지?”

## 7. 메인테이너/관리자 관점에서 관리 비용이 줄어든다
Issue 기반 프로젝트는:
- PR 거절 기준이 명확
- 논의는 코드 밖에서 해결
- 결정 기록이 남음
